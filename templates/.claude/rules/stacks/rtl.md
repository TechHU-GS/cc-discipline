## RTL 工作纪律

### 硬件思维（不是软件思维）
- **每一行代码都是电路** — 写之前想清楚它综合出来是什么硬件结构
- **区分综合和仿真** — `initial`, `$display`, `#delay` 只用于 testbench，永远不出现在可综合代码中
- **时序意识** — 修改任何寄存器逻辑时，考虑对时序路径的影响
- **复位策略一致** — 同步/异步复位不要在同一模块内混用，除非有明确理由

### 修改前强制检查
- 确认修改的信号的扇出（fan-out）— 谁在读这个信号？
- 确认时钟域 — 是否涉及跨时钟域（CDC）？需要同步器吗？
- 确认 bit width — 位宽是否匹配，有没有隐式截断或扩展？
- 确认 FSM 状态 — 修改状态机时，所有状态转移是否完整覆盖？

### 禁止事项
- 禁止用 `always @(*)` 中写锁存器（latch）除非有明确意图和注释
- 禁止在组合逻辑中遗漏 else / default 分支
- 禁止修改接口信号（port）而不同步更新所有例化处
- 禁止在不理解 timing constraint 的情况下修改关键路径逻辑

### Testbench 规则
- 新功能必须有对应的 testbench case
- 修改 RTL 后，先跑现有仿真再看结果
- assertion / checker 优先于波形肉眼检查
